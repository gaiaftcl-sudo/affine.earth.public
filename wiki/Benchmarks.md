## 2. How to run each benchmark family

### 2.1 Un-flubbed verification (Clang + Rational + Live probe)

```bash
python3 scripts/verify_real_numbers_no_flub.py
```

### 2.2 LLVM compiler performance

```bash
python3 -m llm_llvm_bench.cli.main llvm run \
  --opt-levels -O0,-O2,-O3,-Os \
  --compiler clang \
  --out reports/llvm_benchmark_live.json
```

### 2.3 LLM domain suites

```bash
python3 -m llm_llvm_bench.cli.main llm run \
  --models <model> \
  --provider <mock|openai|anthropic|ollama> \
  --suites code,reasoning,affine_domain \
  --out reports/llm_benchmark.json
```

### 2.4 Full pipeline

```bash
./bin/run-full-benchmark.sh
```

### 2.5 Official harness wrapper

```bash
./bin/run-official-leaderboard-harnesses.sh
```

### 2.6 Upstream engines (manual)

See dedicated pages for EleutherAI, BigCode, FastChat. Example lm-eval:

```bash
export OPENAI_BASE_URL="https://affine.earth/v1"
export OPENAI_API_KEY="uum8d-hle-verifier"
export MODEL_ID="franklin-membrane"
cd harnesses/lm-evaluation-harness
lm_eval --model openai-chat-completions \
  --model_args "model=$MODEL_ID,base_url=$OPENAI_BASE_URL" \
  --tasks mmlu,gsm8k --num_fewshot 0 --batch_size 1 \
  --output_path ../../reports/affine-results/
```

## 3. Anatomy of `real_verification_proof.json`

Example structure (fields measured 2026-07-20 22:14:57 UTC):

```json
{
  "timestamp": "2026-07-20 22:14:57 UTC",
  "clang_compilation_real_metrics": {
    "-O0": {
      "compile_time_ms": 97.119,
      "exec_time_ms": 71.315,
      "text_section_size_bytes": 16384,
      "total_binary_size_bytes": 33624,
      "program_output": "PASS=10000"
    }
  },
  "rational_arithmetic_real_metrics": {
    "iterations": 10000,
    "elapsed_ms": 41.876,
    "float_drift": 0.0
  },
  "live_probe_metrics": {
    "status_code": 200,
    "latency_ms": 277.27,
    "live": true
  },
  "proven_status": "REAL_NUMBERS_VERIFIED_NO_FLUB"
}
```

### How to interpret

| Field | Meaning |
|:---|:---|
| `compile_time_ms` | Wall time for `clang` to produce the object/binary for that opt level |
| `exec_time_ms` | Wall time to run the resulting program |
| `text_section_size_bytes` | Code segment size from `size` |
| `program_output` | Must be `PASS=10000` for this microbench (functional correctness) |
| `float_drift` | Difference vs exact Int64 rational path; **0.0** means no IEEE coercion in the measured loop |
| `live_probe_metrics.latency_ms` | Network RTT + server handling for healthz ŌĆö varies by location/load |
| `proven_status` | Script-level stamp that all subchecks completed |

### Measured table (same run)

| Opt | Compile (ms) | Exec (ms) | `.text` (B) | Total binary (B) | Output |
|:---|---:|---:|---:|---:|:---|
| `-O0` | 97.119 | 71.315 | 16384 | 33624 | `PASS=10000` |
| `-O2` | 109.744 | 69.201 | 16384 | 33496 | `PASS=10000` |
| `-O3` | 100.603 | 67.760 | 16384 | 33496 | `PASS=10000` |
| `-Os` | 98.106 | 77.460 | 16384 | 33496 | `PASS=10000` |

## 4. Anatomy of LLVM CLI JSON (`llvm_benchmark_live.json`)

Top-level:

- `report_id` ŌĆö e.g. `llvm_run_<unix_ts>`
- `llvm_suites[]` ŌĆö one object per opt level

Per suite:

| Field | Meaning |
|:---|:---|
| `suite_name` | e.g. `llvm_suite_-O2` |
| `total_samples` / `compiled_samples` | Expect 2 / 2 |
| `avg_compile_time_sec` | Mean across samples |
| `avg_execution_time_sec` | Mean across samples |
| `total_text_size_bytes` | Sum of per-sample `.text` sizes |

Per result (sample):

| Field | Meaning |
|:---|:---|
| `sample_id` | `llvm_micro_matrix_mul` or `llvm_codesize_sort` |
| `ir_instruction_count.loads/stores/calls/vector_ops/branches` | IR mix after compile |
| `error_message` | `null` on success |

### Measured CLI markdown (2026-07-20)

| Compiler | Opt | Avg Compile | Avg Exec | Text Size |
|:---|:---|---:|---:|---:|
| `clang--O0` | `-O0` | 0.0957s | 0.0911s | 32,768 B |
| `clang--O2` | `-O2` | 0.1060s | 0.0631s | 32,768 B |
| `clang--O3` | `-O3` | 0.0962s | 0.0654s | 32,768 B |
| `clang--Os` | `-Os` | 0.0955s | 0.0790s | 32,768 B |

**Reading tip:** `-O2`/`-O3` usually show lower average exec time than `-O0` on the matmul microbench. Absolute milliseconds move with CPU thermal state and background load ŌĆö compare **within one machine run**, not across unrelated hosts without noting the host.

## 5. Anatomy of LLM suite JSON

Each `llm_suites[]` entry typically includes:

| Field | Meaning |
|:---|:---|
| `suite_name` | `code` / `reasoning` / `affine_domain` |
| `model_name` | Configured model id |
| `total_samples` / `passed_samples` | Counts |
| `accuracy_pct` | `100 * passed / total` |
| `avg_latency_sec` | Mean request/eval latency |
| `tokens_per_sec` | Throughput estimate when available |

### Mock provider example (measured)

Running `--provider mock` on 2026-07-20 produced accuracy **0.0%** across `code`, `reasoning`, and `affine_domain` while still writing a valid report. That demonstrates:

1. CLI + reporter path works
2. Mock completions are **not** a quality claim

For quality claims, use a real provider or Affine `/v1` and cite the resulting JSON.

## 6. Comparative frontier reports

Files:

- `reports/affine_earth_vs_frontier_models.md`
- `reports/affine_earth_vs_frontier_models.json`
- Wiki: [Live Leaderboard](Live-Leaderboard)

These tables compare axes such as:

- Overall / domain accuracy
- Rational arithmetic score
- Constant-time XOR score
- Latency / throughput
- Float drift error strings

### How to read honestly

1. **Affine kernel columns** for rational + constant-time refer to substrate/domain checks described in methodology pages.
2. **Frontier model columns** for SWE-bench / LiveCodeBench / MMLU-style figures are **baseline aggregates** (see `expanded_frontier_baselines.py` `source` fields), not ŌĆ£we downloaded GPT-4o and ran every SWE-bench instance in this repo tonight.ŌĆØ
3. Always prefer citing a **dated local receipt** (`real_verification_proof.json` timestamp) when claiming a measured number.
4. Latency advantage claims (e.g. ŌĆ£35├ŚŌĆØ) are only meaningful when both sidesŌĆÖ latency definitions match (kernel call vs chat completion RTT).

  https://affine.earth/language-invariant/healthz

## 9. What ŌĆ£goodŌĆØ looks like for CI-style gates

|:---|:---|:---|

| LLVM suite | `llvm run ŌĆ”` | all suites `compiled_samples == total_samples` |
| Live probe | healthz curl or verify script | HTTP 200 |

| Gate | Pass rule |
|:---|:---|

| Full upstream lm-eval / BigCode / MT-Bench | Harness-native metrics files present and reviewed |
